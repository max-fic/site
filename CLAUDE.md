# CLAUDE.md — Personal Knowledge Base (MkDocs)

A personal **knowledge base and blog**

Goals:
- Organize and publish personal notes, guides, and research
- Support discoverability via search, tags, and a blog section
- Keep the setup simple, fast, and maintainable long-term

---

## Tech Stack

| Layer        | Tool / Library                          |
|-------------|------------------------------------------|
| Site engine  | MkDocs                                  |
| Theme        | Material for MkDocs                     |
| Deployment   | GitHub Pages via GitHub Actions         |
| Extras       | `mkdocs-material[imaging]`, `mkdocs-blog`, built-in search, tags |

## Local Development

* Activate virtual environment for `mkdocs`
```bash
workon mkdocs
```

* Serve locally with live reload
```bash
mkdocs serve
```

* Build static site
```bash
mkdocs build --strict
```

* Deploy Website to GitHub
```bash
mkdocs gh-deploy
```


> **Best practice:** Always use `mkdocs build --strict` before committing — it turns warnings into errors and catches broken links early.

## Content Guidelines

### General
- All content lives under `docs/`. Never write content outside this directory.
- Use **Markdown** only. Avoid raw HTML unless strictly necessary.
- Prefer short, focused pages over long monolithic ones.
- Every page should have a clear `# H1` title at the top.

### File & Folder Naming
- Use `kebab-case` for all file and folder names (e.g. `my-topic/getting-started.md`).
- Name the index page of a section `index.md` — Material will use it as the section landing page.

### Front Matter
Every page should include YAML front matter:

```yaml
---
title: Page Title
description: One-sentence summary for SEO and search.
tags:
  - python
  - tools
---
```

Blog posts must also include:
```yaml
---
date: YYYY-MM-DD
authors:
  - <your-name>
---
```

### Admonitions (use liberally)
```markdown
!!! tip "Pro tip"
    Use admonitions to highlight key information.

!!! warning
    Warn readers about common mistakes.

!!! note
    Add context without interrupting the flow.
```

---

## Tagging Strategy

- Keep a consistent, flat tag vocabulary — avoid near-duplicates like `python` and `Python`.
- Aim for 2–5 tags per page.
- Use tags for **topic** (e.g. `python`, `linux`, `networking`) not for **content type** (avoid tags like `article` or `note`).
- The tags index is auto-generated by the `tags` plugin — no manual maintenance needed.

---

## Blog Guidelines

- Posts live in `docs/blog/posts/`.
- File names should be `YYYY-MM-DD-slug.md` (e.g. `2025-04-23-my-first-post.md`).
- Keep the excerpt short (1–2 sentences) and add `<!-- more -->` to define the fold.
- Blog posts are separate from the knowledge base — don't cross-reference them in the nav.

## What Claude Should and Shouldn't Do

### ✅ Do
- Add or edit Markdown content under `docs/`
- Read and edit files under `specs/` to refine and evolve project specifications
- Read and edit `REQUIREMENTS.md` to capture and refine project requirements
- Read and edit theme template files under `overrides/`
- Suggest improvements to `mkdocs.yml` configuration
- Help write or format blog posts and knowledge base pages
- Add or refine front matter, tags, and admonitions
- Write or fix GitHub Actions workflows
- Use Git for all file changes (see Git Workflow below)

### ❌ Don't
- Modify files outside `docs/`, `specs/`, `overrides/`, `mkdocs.yml`, `.github/`, `REQUIREMENTS.md`, and `requirements.txt`
- Generate raw HTML inside Markdown unless no Markdown alternative exists
- Add plugins not already listed without flagging it first
- Invent tags not consistent with the existing tag vocabulary
- Make changes without committing them via Git

## Git Workflow

All changes must be managed through Git. Claude should follow these rules:

- **Always work on a branch** — never commit directly to `main`
- **Branch naming:** use `feat/`, `fix/`, or `docs/` prefixes (e.g. `feat/add-python-section`, `fix/broken-nav`, `docs/update-requirements`)
- **Commit often** — one logical change per commit, not bulk commits
- **Commit message format:** use [Conventional Commits](https://www.conventionalcommits.org/):
  ```
  feat: add python fundamentals section
  fix: correct broken link in index.md
  docs: update REQUIREMENTS.md with blog goals
  chore: pin mkdocs-material to 9.5.18
  ```
- **Never force-push** to `main` or any shared branch

- **Do not commit** `site/`, `.venv/`, or any auto-generated files — ensure `.gitignore` covers these
